# A-Society: Project Log

## Scope Tags

| Tag | Meaning |
|---|---|
| `[S]` — Small — single-session maintenance change (one to two files) |
| `[M]` — Medium — multi-step change within a single phase |
| `[L]` — Large — multi-phase or multi-file strategic change |
| `[LIB]` — Library — creates or modifies `general/` content; scope additions require Owner approval; may require a framework update report |
| `[MAINT]` — Maintenance — a-docs change within Curator authority; no Owner proposal needed |
| `[ADR]` — Direction decision involved; requires Owner-level scope |
| `[RUNTIME]` — Runtime-root executable change; follows Executable Dev workflow when implementation is required |

Tags may be combined: `[M][LIB]`, `[S][MAINT]`, `[L][LIB][ADR]`.

## Current State

**Status:** Framework active at v34.0. Standing executable model remains fully consolidated: `runtime/` is the sole executable root with the operator surface provided as a local web server and browser UI (project selector, Owner bootstrap chat, React Flow workflow graph). The executable layer now has four developer roles: Framework Services Developer, Orchestration Developer, and UI Developer (newly split from OD), plus the Technical Architect as design authority. Role surface decomposition guidance has been added to the general complexity instruction.

**Recent Focus**

`[S][LIB][ADR]` — **curator-scope-routing** (2026-04-25): Expanded Curator direct authority over `general/` by change type. Clarifications, precision fixes, and worked examples within existing `general/` structure now fall within Curator direct authority; only new scope additions — new instruction types, role templates, or structural categories — require an Owner proposal before implementation. Updated `$A_SOCIETY_WORKFLOW` Shared-Library Approval Invariant and `curator-work` node language, `$A_SOCIETY_CURATOR_IMPL_PRACTICES`, and `$GENERAL_CURATOR_IMPL_PRACTICES`. No update report (human direction). Record: none.
